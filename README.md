# Microcontroladores 2026

Este repositório contém o material da disciplina **Microcontroladores 2026**.

Ao longo da disciplina, este espaço será utilizado para organizar e disponibilizar:

- apresentações das aulas;
- materiais de apoio;
- exemplos de código;
- roteiros e orientações;
- atividades práticas a serem desenvolvidas pelos alunos.

## Organização do repositório

O repositório está estruturado em pastas de acordo com os conteúdos e atividades da disciplina.

De modo geral, poderão estar disponíveis neste repositório:

- **apresentações**: slides e materiais utilizados em aula;
- **aulas**: conteúdos organizados por tema ou prática;
- **exemplos**: códigos de referência para estudo;
- **práticas**: atividades que deverão ser implementadas pelos alunos;
- **recursos**: arquivos auxiliares, modelos e materiais complementares.

## Atividades práticas

As atividades práticas estarão organizadas em pastas específicas.

Cada prática conterá sua própria estrutura interna, incluindo arquivos de apoio, código-fonte e orientações de desenvolvimento.

As **instruções detalhadas de cada atividade** deverão ser consultadas no arquivo `README.md` da respectiva pasta da prática.

## Objetivo do uso do GitHub na disciplina

O uso do GitHub nesta disciplina tem como objetivos:

- organizar os materiais e códigos desenvolvidos ao longo do semestre;
- registrar a evolução das soluções implementadas;
- incentivar boas práticas de versionamento;
- facilitar o acompanhamento das atividades pelo professor;
- manter um histórico claro das entregas realizadas.

## Padronização de uso do Git e do GitHub

Para manter os repositórios organizados, todos os estudantes devem seguir as orientações abaixo.

### 1. Estrutura do repositório

Cada atividade deve ser desenvolvida **na pasta correspondente**, sem criar estruturas paralelas fora do padrão definido no template.

### 2. Nomes de arquivos e pastas

Utilize nomes simples e padronizados:

- prefira letras minúsculas;
- use hífen ou sublinhado quando necessário;
- evite espaços;
- evite acentos e caracteres especiais em nomes de arquivos e pastas.

Exemplos adequados:

- `main.c`
- `README.md`
- `ex01_saida_pino`
- `display_7_segmentos`

### 3. Código-fonte

O código principal de cada exercício deve ser salvo, preferencialmente, dentro da pasta `src/`, conforme a estrutura fornecida no repositório.

### 4. Documentação da atividade

Cada prática possui um arquivo `README.md` próprio. Esse arquivo deve ser preenchido com informações como:

- objetivo da atividade;
- descrição da solução;
- observações de implementação;
- dificuldades encontradas;
- evidências solicitadas pelo professor, quando aplicável.

### 5. Commits

Os commits devem ser feitos ao longo do desenvolvimento, e não apenas ao final da atividade.

Recomenda-se que cada commit registre uma alteração coerente e identificável.

Exemplos de mensagens adequadas:

- `Adiciona estrutura inicial da prática 01`
- `Implementa acionamento de pino de saída`
- `Corrige leitura do botão com pull-up`
- `Atualiza README da prática de GPIOs`

Evite mensagens genéricas como:

- `update`
- `teste`
- `corrigido`
- `versao final`

### 6. Frequência de envio ao GitHub

Após alterações importantes, o estudante deve enviar as atualizações para o repositório remoto com `git push`, mantendo o histórico da atividade atualizado no GitHub.

### 7. Integridade do histórico

Não é permitido apagar o histórico do repositório, recriar o projeto do zero para substituir o anterior ou remover commits para ocultar o processo de desenvolvimento.

O histórico faz parte do acompanhamento da disciplina.

### 8. Entregas

Uma atividade será considerada adequadamente registrada no repositório quando:

- estiver na pasta correta;
- contiver o código-fonte solicitado;
- possuir a documentação mínima exigida no `README.md`;
- tiver sido enviada ao GitHub dentro do prazo estabelecido.

## Recomendação de fluxo de trabalho

Sugere-se o seguinte fluxo básico para cada atividade:

1. acessar a pasta da prática;
2. ler atentamente o `README.md` da atividade;
3. desenvolver o código na estrutura indicada;
4. testar e registrar evidências, quando solicitado;
5. atualizar a documentação da atividade;
6. realizar commits com mensagens descritivas;
7. enviar as alterações para o GitHub.

## Observações finais

Este repositório deverá ser mantido organizado ao longo de toda a disciplina.

A estrutura fornecida tem como objetivo facilitar o desenvolvimento das atividades, a documentação das soluções e o acompanhamento do progresso dos estudantes.

Em caso de dúvida, o aluno deve consultar primeiro o `README.md` da prática correspondente e, se necessário, entrar em contato com o professor.
